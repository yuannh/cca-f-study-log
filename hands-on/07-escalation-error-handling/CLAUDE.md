# 动手项目7：升级(Escalation)标准 + 错误传播策略设计

**对应计划**：第3周 周五（2026-07-24）| **官方 Domain**：5 上下文管理与可靠性（Task Statement 5.2、5.3）

## 任务目标

不需要完整代码实现，写一份设计文档（`design.md`），针对一个客服/多智能体场景，明确定义：什么情况下应该升级给人工，什么情况下不同类型的错误应该如何在多智能体系统中传播。

## 验收标准

- [ ] 升级标准是具体、可判断的规则（例如"客户明确要求人工"、"政策对这个情况没有覆盖"、"连续N次尝试仍无进展"），而不是"看起来很复杂就升级"这种模糊标准
- [ ] 明确说明：情绪化的语气 / 模型自己汇报的置信度分数，为什么不是可靠的升级触发信号
- [ ] 区分至少3类错误：瞬时错误(可重试)、业务规则错误(不可重试，需说明原因)、合法的空结果(不是错误)
- [ ] 设计的错误传播格式包含：失败类型、已尝试的操作、部分结果——而不是一个笼统的"失败"状态
- [ ] 举一个具体反例：为什么"任何异常都终止整个工作流"和"任何异常都静默吞掉继续跑"都是反模式
